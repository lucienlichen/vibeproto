# 更新与部署工作流

## 概览

本项目采用 **GitHub Actions 自动化 CI/CD**，推送代码到 `main` 分支后，自动完成镜像构建、推送和服务器部署，全程无需手动操作。

```
本地开发 → git push → GitHub Actions → 构建 Docker 镜像 → 推送 GHCR → SSH 部署服务器
```

## 日常开发更新流程

### 1. 修改代码（前端或后端）

```bash
# 前端开发（本地预览）
cd frontend
npm run dev         # 访问 http://localhost:5173

# 后端开发（本地运行）
cd backend
./mvnw spring-boot:run
```

### 2. 本地验证构建（推送前必做）

前端有 TypeScript 类型检查，本地先跑一遍确保无报错：

```bash
cd frontend
npm run build       # 必须通过，否则 CI 会失败
```

后端编译检查：

```bash
cd backend
./mvnw package -DskipTests -q
```

### 3. 提交并推送到 main

```bash
git add .
git commit -m "描述本次改动"
git push origin main
```

推送后 **GitHub Actions 自动触发**，无需任何额外操作。

### 4. 观察部署进度

打开 https://github.com/lucienlichen/vibeproto/actions

流程分两个 Job：

```
build-and-push（约 3-5 分钟）
  ├─ 构建 backend Docker 镜像
  ├─ 构建 frontend Docker 镜像
  └─ 推送到 GHCR（ghcr.io/lucienlichen）

deploy（约 1 分钟，build-and-push 成功后自动触发）
  ├─ 同步 deploy/ 配置文件到服务器
  ├─ 服务器拉取新镜像
  └─ docker compose up -d 重启服务
```

全部绿色 ✓ 后，访问 http://106.54.16.240 即可看到更新。

## 仅更新前端

只改动了 `frontend/` 目录下的文件，流程完全相同——push 即触发全量构建。
前后端镜像会各自独立构建，互不影响。

## 仅更新部署配置

只改动了 `deploy/` 目录（如 `docker-compose.yml`、`nginx.conf`）：

push 后 CI 会重新构建镜像（耗时较长），如果只想快速生效配置，可直接 SSH 到服务器手动同步：

```bash
ssh root@106.54.16.240

# 手动修改配置
vim /opt/vibeproto/deploy/nginx/nginx.conf

# 重启对应服务
cd /opt/vibeproto/deploy
docker compose restart prototype-server   # 只重启 nginx
# 或
docker compose up -d                      # 重启所有
```

## 服务器运维命令

SSH 登录服务器：

```bash
ssh root@106.54.16.240
cd /opt/vibeproto/deploy
```

常用命令：

```bash
# 查看所有容器状态
docker compose ps

# 查看某个服务日志
docker compose logs -f backend       # 后端日志
docker compose logs -f frontend      # 前端日志
docker compose logs -f prototype-server  # Nginx 日志

# 手动重启某个服务
docker compose restart backend

# 重启所有服务
docker compose up -d

# 查看磁盘占用
df -h
du -sh /opt/vibeproto/projects/*     # 各项目占用空间
```

## CI/CD 失败排查

### 构建失败（build-and-push job）

**前端编译报错（最常见）**：

本地先执行 `cd frontend && npm run build`，修复所有 TypeScript 报错后再推送。

**后端编译报错**：

本地执行 `cd backend && ./mvnw package -DskipTests`，确保无错误。

### 部署失败（deploy job）

**SSH 连接失败**：检查 GitHub Secrets 中 `SERVER_HOST`、`SERVER_USER`、`SERVER_SSH_KEY` 是否正确。

**镜像拉取失败**：可能是 GHCR 认证问题，在 Actions 日志中查看具体报错。

**容器启动失败**：SSH 到服务器查看日志：

```bash
docker compose logs backend --tail=50
```

### 回滚到上一个版本

```bash
ssh root@106.54.16.240
cd /opt/vibeproto/deploy

# 查看历史镜像
docker images | grep vibeproto

# 指定旧 tag 启动（将 abc12345 替换为目标 commit 短 SHA）
IMAGE_TAG=abc12345 docker compose up -d backend frontend
```

## 环境变量说明

服务器配置文件位于 `/opt/vibeproto/deploy/.env`，修改后需重启对应服务：

| 变量 | 说明 | 修改后需重启 |
|------|------|------------|
| `DB_PASS` | MySQL 密码 | mysql + backend |
| `JWT_SECRET` | JWT 签名密钥（修改后所有用户需重新登录） | backend |
| `BASE_PREVIEW_URL` | 原型预览基础地址 | backend |
| `MINIO_SECRET_KEY` | MinIO 密码 | minio + backend |

修改 `.env` 后重启：

```bash
cd /opt/vibeproto/deploy
docker compose up -d
```
